## 카톡 초안 (간결 버전)

안녕하세요 조교님, 지난 미팅에서 말씀하신 "부품을 3D 좌표값으로 뽑을 수 있는지 확인" 건 사전 보고드립니다.

저희 CAD(STL)로 먼저 가능성을 확인해봤는데, **27종 전부 추출 가능**했습니다. (28종 중 1종은 보유 프린터 빌드볼륨 초과로 실물 출력이 안 돼 데이터셋에서 제외 → 27종으로 통일했습니다.)

- 27종 모두 watertight라 **부피중심(COM)·관성 주축·안정자세**가 깨끗이 나옵니다.
- 그래서 부품마다 **위치 기준점(COM) + 자세 기준축(주축 쿼터니언) + canonical bbox + 키포인트 14개 + grasp 폭**까지 산출해뒀습니다.

다만 두 가지를 정하고 싶어 여쭙니다:

1. 여기서 **"3D 좌표"가 어떤 의미**일까요?
   - ① 부품별 키포인트 (학습 타깃)
   - ② 6DoF pose (COM 위치 + 주축 회전)
   - ③ grasp 6요소 (x,y,z,edge,angle,label)
   → 어느 쪽이냐에 따라 정규화 기준을 맞추려고 합니다.

2. STL 다수가 **메시 원점이 형상 밖**(최대 1m)이라 그대로 쓰면 안 돼서, **COM을 원점으로 재정규화**했습니다. 이 기준이 조교님 모델(3D 인코더 pointcloud)과 정합되는지 확인 부탁드립니다.

산출 좌표(JSON)는 정리되는 대로 같이 공유드리겠습니다.

## 근거 데이터 (내부 — 메시지엔 표만 첨부 가능)

### 추출 가능성 (probe_3d_coords.py)
- 27/27 watertight → COM·주축·안정자세 신뢰 추출
- 등방축(자세 모호) 부품 0종
- ⚠️ 원점 offset>50mm = 22/27 → 좌표계 정규화 필요 (해결함)
- ⚠️ 길쭉/평판(top_inner_sheet 안정자세 64, bracket_sensor2 anis 41) → canonical frame 부호 규칙 필요 (해결함)

### 정규화 산출 (probe_3d_coords_normalized.py → coords_27parts.json)
- 27종 전부: COM(world mm) + 주축 quat(wxyz) + canonical bbox + 키포인트 14개 + grasp 폭
- grasp 최소폭<40mm = 25/27종 (대부분 일반 그리퍼 가능)

### 미해결 / 조교 확인 필요
- "3D 좌표"의 정의 (키포인트 / 6DoF / grasp 6요소)
- COM 원점 정규화가 3D 인코더(pointcloud 패치)와 정합되는지
- canonical frame 부호 규칙이 좌우대칭 부품 쌍(l/r)에서 일관적인지 (추가 검증 여지)
