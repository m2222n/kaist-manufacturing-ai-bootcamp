# 조교 제공 코드 리뷰 — CADENCE 파이프라인 (2026-06-23)

> 대상: `github.com/LimHaksoo/Mentoring` (커밋 `9972f2a`). `~/kaist_project/model/`에 통합.
> 목적: 구조 이해 + 우리 데이터 정합 + 비교 실험·발표 전 알아둘 점 정리.
> 검증: A100에서 split→pc→3D enc→memory bank→detector warmup→joint→eval 전 과정 직접 구동하며 리뷰.

---

## 1. 전체 구조 — 한눈에

| 단계 | 파일 | 역할 |
|------|------|------|
| pointcloud | `stl_to_pointcloud_dataset.py` | STL → 8192점(xyz+normal), edge 샘플링 비율·sharp angle 옵션 |
| 3D encoder | `train_cad_encoder.py` + `cad3d_encoder/`, `pointnet2_encoder.py` | **PointNet++** set abstraction, 분류+contrastive 학습 |
| codebook | `build_cad_memory_bank.py` | 27 CAD 임베딩 (전역 256-d) + local token (256×256) 저장 |
| 2D detector | `train_depth_vq_detector.py` + `depth_vq_detector/` | **DETR/Mask2Former 계열** depth-only query detector + CAD VQ head |
| 평가 | `eval_depth_vq_detector.py` | Hungarian 매칭 기반 recall/class/box/mask/**cad** acc |
| 추론 | `infer_depth_vq_detector.py` | depth-only 입력 → box/mask/class/cad_id + 시각화 |

**모델 본체(`depth_vq_detector/model.py`)**: DepthCNN backbone → FPN pixel decoder(mask feat + transformer memory) → Transformer object decoder(쿼리 100, 6층, 8헤드) → 4 헤드(class / box / mask / **CAD VQ**). CAD VQ head = object embedding을 **frozen CAD codebook과 cosine 유사도** → cad_logits.

## 2. 강점 (잘 만든 점)

1. **우리 데이터 포맷에 정확히 맞춤.** scene npz의 `depth/inst_id/category_id/meta` 키, 배경 NaN, category 1..27, `label_offset=1`까지 우리 6/17 1000장과 **완전 일치**. 조교가 우리 5장 샘플 포맷을 그대로 받아 작성 → 추가 변환 0.
2. **leakage 방지가 설계에 박혀 있음.** `tools/make_scene_splits.py`가 **scene 단위** split(crop 단위 금지 명시) + bg_kind stratify + seed 고정. README가 crop 단위 split의 leakage 위험을 직접 경고.
3. **inference가 depth-only임을 강제.** `model.forward(x)`는 depth 채널만 받음. `inst_id/category_id/meta`는 GT 매칭·loss·시각화에만 쓰이고 forward에 안 들어감 → 실제 배포(depth 카메라 1대)와 동일. README §10에 depth-only 파일로 검증하는 절차까지 제공.
4. **CAD 이름 매칭이 견고함.** `canonical_cad_aliases()`가 `03_sol_block_front.stl` ↔ `03_sol_block_front` ↔ pointcloud의 hash suffix(`__e79f118d`) ↔ 경로/소문자까지 정규화. 우리 STL 이름 컨벤션과 자동 호환.
5. **2단계 학습(det warmup → joint).** CAD VQ가 초기에 노이즈일 때 detection이 먼저 안정화되도록 분리. `score_mode`(det/product/cad)로 단계별 디버깅 가능.
6. **의존성 최소.** `requirements = numpy/torch/tqdm`. scipy 없으면 greedy 매칭으로 fallback(스모크 테스트용). backbone도 의존성 없는 자작 CNN(나중에 ConvNeXt 등으로 교체 가능하게 API 유지).
7. **표준 DETR/Mask2Former 손실 정석.** Hungarian matcher + class CE(no-object eos_coef=0.1) + box(L1+GIoU) + mask(BCE+Dice) + CAD(CE+cosine align). matcher cost와 loss weight 분리.

## 3. 우리 데이터·환경 정합 (실측)

- 3D encoder: 27 CAD 분류·식별 정확도 **1.0** 도달(58 epoch경), codebook (27, 256) 정상 저장. → 27종이 형상 임베딩 공간에서 완전 분리됨.
- memory bank: 전역 임베딩 + local token(256×256) 둘 다 저장(`--save_local`). detector는 전역 embeddings만 사용.
- detector: `--num_workers 0` 필수(A100 `/dev/shm` 64MB 제약, 안 그러면 DataLoader OOM). 1000 scene이라 단일 프로세스 로딩이 약간 느리지만 동작엔 문제 없음.

## 4. 주의점 / 잠재 이슈 (비교 실험·발표 전 인지)

1. **데이터 규모가 작다.** scene 1000장(train 800), 부품 27종. DETR 계열은 보통 큰 데이터에서 진가 → 절대 성능보다 **파이프라인 동작 입증** 단계로 봐야 함. (조교도 별도 고도화 중.) → 발표에서 "중간 결과"로 정직하게 표기.
2. **`best.pt` 선택 기준 = val loss_total.** mask/box mAP가 아니라 loss 총합 최소. 빠른 실험엔 OK지만, 정식 비교 땐 recall/mAP 기준 선택도 고려.
3. **AMP 기본 off.** `--amp` 줘야 mixed precision. A100이면 켜서 속도·메모리 이득 가능(수치 미세 차이 주의).
4. **`num_queries=100` vs 실제 부품 수.** 우리 scene은 visible 부품이 보통 8~14개. 쿼리 100개는 충분하지만, 매우 밀집한 적재형에선 상한 확인 필요.
5. **clip_grad_norm=0.1로 강함.** 초기 발산 억제용으로 보수적. 수렴 느리면 완화 여지.
6. **scipy 없으면 greedy 매칭.** 정식 실험은 scipy 설치 확인(A100엔 있음 → 정확 Hungarian 사용 중).
7. **공개 데이터셋(BOP/ITODD) 포맷 차이.** 이 코드는 우리 npz 포맷 전용 로더. BOP/ITODD는 BOP 포맷(scene_gt.json, depth png, models) → **변환 어댑터를 우리가 작성**해야 비교 가능(검출 GT를 inst_id/category_id 맵으로 변환). 이게 비교 실험의 핵심 작업량.
8. **카메라 intrinsic.** `xyzv/xyznv` 모드는 intrinsic 필요. 우리는 `zv`로 시작(정합). 실물 Depth 카메라 비교 땐 intrinsic 채우면 `xyzv` 시도 가능.
